## Architecture Compliance

### Core Decisions Relevant to This Story

**Decision 1: Static Site Generation Engine**
- **Choice:** 11ty (Eleventy) v3.1.0
- **Why:** Zero runtime dependencies, cPanel-compatible, native JSON data support
- **Implementation:** Use official minimal template to establish foundation
- **Affects:** This story + all subsequent build and deployment stories

[Source: architecture.md#Starter Template Evaluation, lines 50-75]

**Decision 2: Code Organization Pattern**
- **Choice:** Modular template + data file architecture
  - Templates in `src/`
  - Reusable components in `src/_includes/`
  - Global data in `src/_data/`
  - Build output to `_site/`
- **Why:** Separates concerns, scales well, matches 11ty conventions
- **Affects:** How all templates and data are organized in future stories

[Source: architecture.md#Code Organization, lines 105-110]

**Decision 3: Build Output Location**
- **Choice:** Always build to `_site/` directory
- **Why:** Standard 11ty convention, matches cPanel FTP deployment expectations
- **Affects:** GitHub Actions workflow, FTP deployment configuration

[Source: architecture.md#Build Tooling, lines 102-103]

### Architectural Constraints to Respect

1. **cPanel Hosting Limitation:** Project must generate pure static HTML/CSS/JS (no Node.js runtime on hosting)
   - This initialization satisfies this by using 11ty (static generator, not SSR)
   - All build happens locally or in GitHub Actions
   - Output is static files only

[Source: architecture.md#Technical Constraints, lines 34-41]

2. **Data Architecture Compatibility:** Project structure must support properties.json in `src/_data/`
   - Do NOT create custom directory structures
   - Do NOT use database abstractions
   - Keep data files in standard location for 11ty's auto-loader

[Source: architecture.md#Data Architecture, lines 157-182]

3. **Responsive CSS Foundation:** Project must support Tailwind CSS integration in next steps
   - The minimal template supports npm package integration
   - Do NOT pre-configure CSS; that's a separate story
   - Ensure `_includes/` and `src/` can accept CSS files

[Source: architecture.md#Frontend Architecture, lines 133-139]

## Dev Notes

### Important Context & Patterns

**11ty Minimal Template Characteristics:**
- Very lightweight starter (no theme, no example pages)
- Includes basic configuration but minimal boilerplate
- Expects developer to create templates from scratch (which is fine—we have stories for that)
- Default template language is Nunjucks (powerful template syntax, good for this project)

**Build Process Overview:**
1. `npm run build` calls 11ty CLI
2. 11ty reads `src/` directory
3. Processes templates, applies data from `src/_data/`
4. Outputs HTML files to `_site/`
5. Ready for cPanel FTP deployment (this is the entire deployment unit)

**Development Workflow:**
- Use `npm run dev` during development for instant feedback
- Use `npm run build` for production builds (what GitHub Actions will run)
- Both commands generate identical output (dev just adds server + hot reload)

### Architecture Patterns to Establish Now

1. **Directory Structure Discipline:**
   - ALL templates in `src/` or `src/_includes/` (no random directories)
   - ALL data in `src/_data/` (enables 11ty's auto-loader)
   - ALL output verified in `_site/` before deployment

2. **Data File Naming Convention:**
   - Use camelCase for data files: `properties.json`, `siteConfig.js`
   - Store in `src/_data/` for automatic 11ty loading
   - Enables accessing data in templates without explicit imports

3. **Build Configuration Minimal Philosophy:**
   - Keep `.eleventy.js` simple; only add config when required
   - Avoid custom build plugins until Story 1.3 (image optimization)
   - Let 11ty handle the happy path

### Known Variations from Default Minimal

**Expected Behavior:**
The minimal template should initialize with:
- `src/index.html` (default page)
- `.eleventy.js` with basic config
- `package.json` with 11ty + dependencies
- Hot reload working in dev mode

**If You See Warnings:**
- "no .eleventy-git-ignore file" — safe to ignore, not critical
- Missing some directories — create them if missing (e.g., `src/_data/`)
- npm cache warnings — safe, npm handles cleanup
